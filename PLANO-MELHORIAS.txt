PLANO COMPLETO DE MELHORIAS — VÉRTICE INVEST
================================================================

----------------------------------------------------------------
CONTEXTO (por que este plano existe)
----------------------------------------------------------------
Pedido: um plano de melhoria que cubra TODAS as áreas do projeto,
explicado para quem não programa, com 10 ações por categoria, sem
inventar problema que não existe ("falso positivo"), dizendo a
COMPLEXIDADE e QUAL IA usar (Opus 4.8 ou Sonnet 4.6) para economizar
tokens.

O projeto inteiro foi varrido (backend, frontend e infraestrutura) e
as alegações mais graves foram verificadas à mão. Isso já derrubou 2
falsos positivos:

  [X] "Segredos vazaram no histórico do Git" -> FALSO. O arquivo .env
      nunca foi enviado ao Git. Não há vazamento.
  [!] "useCallback não é usado" -> FALSO. É usado em 5 arquivos.

DIAGNÓSTICO GERAL: o Vértice é um projeto maduro e bem cuidado
(nota geral ~B+). Tem segurança forte (JWT, 2FA, criptografia, rate
limit), CI/CD com GitHub Actions, linter, Husky, documentação boa e
39 testes no backend. Os pontos fracos reais são: poucos testes no
frontend, sem Docker/deploy automatizado, alguns "remendos" de código
e falta de monitoramento de métricas. Não é um projeto em crise — é um
projeto bom que pode virar excelente.

----------------------------------------------------------------
COMO LER ESTE PLANO (legenda)
----------------------------------------------------------------
Cada ação tem dois marcadores.

COMPLEXIDADE (quanto trabalho dá):
  [BAIXA]  algumas horas, mexe em poucos arquivos, baixo risco.
  [MEDIA]  de meio dia a 1 dia, exige cuidado/teste.
  [ALTA]   vários dias, mexe em muita coisa ou em lógica delicada.

QUAL IA USAR (para gastar token só onde precisa):
  [SONNET] Sonnet 4.6 — tarefa mecânica, repetitiva, com padrão claro
           (testes seguindo um modelo, aria-label, atualizar
           dependência, documentação, config). A MAIORIA. Use por padrão.
  [OPUS]   Opus 4.8 — lógica complexa, dinheiro/finanças, arquitetura,
           segurança crítica, bug difícil. Use só quando errar custa caro.

REGRA DE BOLSO: se a tarefa é "siga este padrão e repita" -> Sonnet.
Se é "pense, decida e não pode errar" -> Opus.
